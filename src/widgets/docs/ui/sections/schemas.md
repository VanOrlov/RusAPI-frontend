Сердце **RusAPI** — это гибкая система схем, построенная поверх библиотеки `faker.js`. Схема определяет **структуру** и **тип генерации** для каждого поля вашего ресурса.

Один раз настроили — и сервис сам сгенерирует вам реалистичные данные.

---

## Что такое схема?

Схема — это набор полей с описанием:

- **имени поля** — например, `email`, `fullName`, `avatarUrl`;
- **типа данных** — строка, число, булево и т.д.;
- **генератора (faker‑путь)** — например, `internet.email`, `person.fullName`, `image.avatar`.

После сохранения схемы RusAPI:

1. Генерирует стартовый набор записей (по умолчанию 10 штук).
2. Применяет выбранные генераторы при создании новых сущностей.

---

## Пример схемы для ресурса `users`

Допустим, вы создаете ресурс `users`. В интерфейсе вы добавляете поля:

- `email` — тип: строка, генератор: `internet.email`
- `fullName` — тип: строка, генератор: `person.fullName`
- `avatar` — тип: строка, генератор: `image.avatar`
- `age` — тип: число, генератор: `number.int`

После сохранения схемы стартовые данные могут выглядеть так:

```json
[
  {
    "id": "6c3e3e33-3b2e-4b0f-b9a9-7c513c2d0a91",
    "email": "anna89@example.com",
    "fullName": "Anna Petrova",
    "avatar": "https://example.com/avatar1.png",
    "age": 27
  }
]
```

Точные значения будут отличаться, но **структура** и **типы** останутся такими, как вы указали в схеме.

---

## Поле `id` и системные атрибуты

Поле `id` добавляется автоматически и не редактируется через схему:

- оно всегда имеет уникальное значение;
- формат — UUID v4;
- используется во всех `GET / PATCH / PUT / DELETE` запросах.

Благодаря этому ваши моки ведут себя как настоящий REST‑бэкенд.

---

## Как выбирать генераторы (faker‑типы)?

При настройке поля вы можете выбрать один из множества готовых генераторов:

- **Текст и имена:** `person.firstName`, `person.fullName`, `lorem.sentence`
- **Интернет‑данные:** `internet.email`, `internet.url`, `internet.userName`
- **Изображения:** `image.avatar`, `image.url`
- **Числа и даты:** `number.int`, `number.float`, `date.past`, `date.future`

Рекомендации:

- Для идентификаторов, которые вам не нужны на фронте, можно использовать простые `number.int`.
- Для UI‑демо (карточки профиля, карточки товара) выбирайте «говорящие» типы (`person.fullName`, `commerce.productName` и т.д.).

---

## Обновление схемы на лету

Вы можете менять схему ресурса даже после того, как начали работать с ним:

1. Добавлять новые поля.
2. Удалять ненужные.
3. Менять генератор и тип.

После сохранения:

- новые сущности будут создаваться уже по **обновленной** схеме;
- существующие данные не ломаются — вы сами решаете, пересоздавать ли их.

Такой подход позволяет вам быстро подстраивать моки под эволюцию фронтенд‑контракта.
