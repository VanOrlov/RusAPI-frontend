Как и любой сервис, **RusAPI** имеет ряд ограничений, которые помогают поддерживать стабильность и предсказуемость работы для всех пользователей.

Ниже перечислены типичные лимиты, с которыми стоит считаться при проектировании ваших моков.

---

## Размер выборки и пагинация

При запросе списка ресурсов:

- `GET /api/mock/:nanoId/:resourceName`
- `GET /api/mock/:nanoId/:resourceName?offset=0&limit=20`

обычно действуют следующие правила:

- **Максимальное количество записей за один запрос** — ограничено (например, 100 элементов).
- Если вы не передаёте `limit`, используется значение по умолчанию.
- Параметры `offset` / `page` / `limit` позволяют имитировать поведение реального продакшен‑API.

Рекомендуется:

- на фронте **всегда закладываться на пагинацию**;
- не пытаться вытянуть «всё и сразу» для больших коллекций.

---

## Ограничения по количеству проектов и ресурсов

В зависимости от тарифа/конфигурации могут быть лимиты:

- на количество **проектов** на одного пользователя;
- на количество **ресурсов (эндпоинтов)** в рамках проекта;
- на суммарное число **записей** в одном ресурсе.

Типичный сценарий:

- десятки проектов для разных pet‑проеков и прототипов;
- несколько–десятков ресурсов внутри проекта (`users`, `orders`, `products` и т.д.).

---

## Ограничения по размеру тела запроса

Для запросов `POST`, `PUT`, `PATCH` действует лимит на размер `request body`:

- не отправляйте огромные массивы в одном запросе;
- не пытайтесь «залить» сразу весь тестовый датасет на тысячи строк.

Если лимит превышен, вы можете получить:

- `413 Payload Too Large`;
- либо общий ответ об ошибке валидации / обработки запроса.

---

## Скорость запросов и анти‑DDoS

RusAPI не предназначен для нагрузочного тестирования. Возможны:

- ограничения на количество запросов **в секунду / минуту** с одного IP или токена;
- временные блокировки при агрессивном спаме.

Рекомендации:

- не запускайте бесконечные циклы с частыми `fetch`‑запросами;
- при интеграции с автотестами добавляйте небольшие задержки;
- кэшируйте результаты на фронте, если это возможно.

---

## Как использовать лимиты с пользой

Правильно настроенные ограничения помогают:

- проектировать UI сразу с учётом пагинации и пустых состояний;
- не завязываться на «магические» бесконечные списки;
- не строить логику, критически зависящую от огромных объёмов мок‑данных.

Если при разработке вы регулярно упираетесь в лимиты, это хороший сигнал пересмотреть архитектуру запросов — **так же, как это случилось бы на реальном бэкенде**.
