Чтобы ваши моки были максимально похожи на настоящий бэкенд, **RusAPI** строго проверяет входящие данные по схеме ресурса.

Это значит:

- никакого «мусора» в запросах;
- никаких строк вместо чисел;
- никаких лишних полей, о которых бэкенд «не знает».

---

## Как работает валидация

Когда вы отправляете запрос `POST`, `PUT` или `PATCH`:

1. RusAPI берет сохраненную схему ресурса.
2. Сравнивает **каждое поле** тела запроса с ожидаемым типом.
3. Автоматически:
   - отбрасывает поля, которых **нет в схеме**;
   - проверяет, что типы совпадают;
   - при ошибках возвращает `400 Bad Request`.

---

## Поддерживаемые типы данных

Конкретный набор типов определяется конфигурацией сервера, но типично доступны:

- **string** — строки (email, имя, URL и т.д.);
- **number** — числа (цены, возраст, количество);
- **boolean** — логические значения (`true` / `false`);
- **date / datetime** — даты и время (обычно в ISO‑формате);
- **array / object** — сложные вложенные структуры (если они заданы в схеме).

При несоответствии типов RusAPI не «угадывает» за вас, а честно возвращает ошибку.

---

## Примеры корректных и некорректных запросов

Допустим, у вас есть схема:

- `email` — тип `string`;
- `age` — тип `number`;
- `isActive` — тип `boolean`.

### Корректный запрос

```http
POST https://rusapi.fun/api/mock/:nanoId/users
Content-Type: application/json

{
  "email": "user@example.com",
  "age": 22,
  "isActive": true
}
```

Ответ: `201 Created` и валидный объект пользователя.

### Некорректный запрос

```http
POST https://rusapi.fun/api/mock/:nanoId/users
Content-Type: application/json

{
  "email": "user@example.com",
  "age": "двадцать два",
  "isActive": "yes",
  "extraField": "я не в схеме"
}
```

Возможный ответ:

```json
{
  "statusCode": 400,
  "message": "Validation failed",
  "errors": {
    "age": "Expected number",
    "isActive": "Expected boolean"
  }
}
```

Поле `extraField` будет проигнорировано, даже если тип у него «правильный».

---

## Зачем это нужно фронтенду

- Вы раньше замечаете несоответствия между вашим UI и контрактом API.
- Тестовые данные не превращаются в «свалку» из разнотипных объектов.
- Переход на настоящий бэкенд происходит максимально плавно — контракты уже чистые.

Таким образом, RusAPI помогает не только «показывать какие‑то данные», но и дисциплинирует структуру запросов.
